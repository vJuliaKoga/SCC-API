切り分け結果として、API 側は動いていて、問題は Keploy の録画条件です。WSL では `keploy 3.3.62` が見えていて、BFF も `rest-backend` に向けて正常起動し、`/health` と `/api/users/1` は返りました。

今回の失敗原因は、`sudo -E keploy record ...` で Keploy が root 権限でアプリを起動した結果、Gradle のキャッシュ参照先が `/home/testuser/.gradle` ではなく `/root/.gradle` に切り替わったことでした。そのため、Keploy 実行中だけ Gradle が依存を再ダウンロードしにいき、`plugins.gradle.org` へのアクセスで `Read timed out` になっていました。

その結果、[bff/keploy/test-set-0/mocks.yaml](/c:/Users/juria.koga/Documents/Github/SCC-API/bff/keploy/test-set-0/mocks.yaml) には API テストケースではなく、Gradle の外部通信や起動ログが大量に記録されていました。

設定は [bff/keploy.yml](/c:/Users/juria.koga/Documents/Github/SCC-API/bff/keploy.yml#L5) を直して、`command` で `GRADLE_USER_HOME` を元ユーザーのホーム配下へ固定するようにしました。あわせて [buildDelay](/c:/Users/juria.koga/Documents/Github/SCC-API/bff/keploy.yml#L18) も `50` に上げています。


もしこれでも `mocks.yaml` しか出ないなら、次の本命は `bootRun` ではなく `java -jar` 起動へ寄せることです。そこは試しましたが、現状 `./gradlew bootJar` が失敗するので、次はそのビルド修正を一緒にやるのがよさそうです。

- Keploy install docs: https://keploy.io/docs/concepts/installation/

原因は BFF 本体ではなく、Keploy が `sudo` で起動した Gradle のキャッシュ先です。今回の失敗では root 側の `/root/.gradle` を見に行って依存を再ダウンロードし、`plugins.gradle.org` で `Read timed out` になっていました。だから `BUILD FAILED` になり、Keploy も「アプリが落ちた」と判断して止まっています。巨大化した [mocks.yaml](/c:/Users/juria.koga/Documents/Github/SCC-API/bff/keploy/test-set-0/mocks.yaml) にも、API ではなく Gradle の外部通信が大量に入っていました。
