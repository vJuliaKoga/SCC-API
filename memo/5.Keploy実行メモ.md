## Keploy test の実行メモ

`bff/keploy/test-set-4/tests` が生成されていても、同じ test-set の `mocks.yaml` に何が入っているかで実行方法が変わる。

- `bff/keploy/test-set-4/mocks.yaml` が `localhost:19092` 向けの HTTP mock を持っている場合:
  - これは `app.call-mode=rest` で録画した test-set
  - `app.call-mode=grpc` でそのまま mock 再生すると噛み合わない可能性が高い

### 1. 同じ mode で素直に test する場合

`keploy.yml` の `command` が `rest` 前提なので、まずはこれが最も安全。BFF の起動に 40 秒前後かかるため、`test.delay` は `50` に上げてある。

```bash
cd /mnt/c/Users/juria.koga/Documents/Github/SCC-API/bff
sudo -E keploy test --test-sets test-set-4 --delay 50
```

### 2. BFF だけ grpc mode に切り替えて挙動比較したい場合

この場合は `mocks.yaml` を使わず、実際の `grpc-backend` を起動した状態で test する。

別ターミナルで `grpc-backend` を起動:

```bash
cd /mnt/c/Users/juria.koga/Documents/Github/SCC-API/grpc-backend
./gradlew --no-daemon bootRun
```

その上で `bff` 側を `grpc` mode で起動して test:

```bash
cd /mnt/c/Users/juria.koga/Documents/Github/SCC-API/bff
sudo -E keploy test \
  --test-sets test-set-4 \
  --delay 60 \
  --mocking=false \
  -c "bash -lc 'USER_HOME=\"\$(getent passwd \"\${SUDO_USER:-\$USER}\" | cut -d: -f6)\"; export GRADLE_USER_HOME=\"\${GRADLE_USER_HOME:-\$USER_HOME/.gradle}\"; ./gradlew --no-daemon bootRun --args=\"--app.call-mode=grpc\"'"
```

`base-path` を使うと Keploy は BFF を起動も instrument もしない。

- `mocks.yaml` がある test-set をそのまま流すと `Post "/storemocks": unsupported protocol scheme ""` で落ちることがある
- `mocks.yaml` を外すと、Keploy 3.3.62 では test-set として認識されない挙動があった

このため、`rest` で録った test-set を `grpc` live 実行へ流用するやり方は不安定。

grpc 比較をちゃんとやるなら、`app.call-mode=grpc` で録画し直して grpc 用 test-set を別で持つのが安全。

```bash
# terminal 1
cd /mnt/c/Users/juria.koga/Documents/Github/SCC-API/grpc-backend
./gradlew --no-daemon bootRun
```

```bash
# terminal 2
cd /mnt/c/Users/juria.koga/Documents/Github/SCC-API/bff
sudo -E keploy record \
  --path keploy \
  -c "bash -lc 'USER_HOME=\"\$(getent passwd \"\${SUDO_USER:-\$USER}\" | cut -d: -f6)\"; export GRADLE_USER_HOME=\"\${GRADLE_USER_HOME:-\$USER_HOME/.gradle}\"; ./gradlew --no-daemon bootRun --args=\"--app.call-mode=grpc\"'"
```

別ターミナルで Bruno か curl から `http://127.0.0.1:19090/api/...` を送る。

その後の test:

```bash
cd /mnt/c/Users/juria.koga/Documents/Github/SCC-API/bff
sudo -E keploy test \
  --path keploy \
  --test-sets <grpcで録画したtest-set名> \
  --delay 80 \
  -c "bash -lc 'USER_HOME=\"\$(getent passwd \"\${SUDO_USER:-\$USER}\" | cut -d: -f6)\"; export GRADLE_USER_HOME=\"\${GRADLE_USER_HOME:-\$USER_HOME/.gradle}\"; ./gradlew --no-daemon bootRun --args=\"--app.call-mode=grpc\"'"
```

### 3. grpc mode を Keploy の mock 付きで回したい場合

一番きれいなのは、`app.call-mode=grpc` で録画し直して grpc 用の test-set を別に作ること。

- `rest` 録画の test-set: `test-set-rest`
- `grpc` 録画の test-set: `test-set-grpc`

のように分けると混ざらない。
