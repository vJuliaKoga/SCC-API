
# Observability 環境 利用ガイド

本ドキュメントでは、SCC-API の Observability 構成において  
Grafana / Prometheus / OpenTelemetry Collector をどのように利用するかを整理する。

---

## 環境構成

| コンポーネント | URL |
|----------------|-----|
| Grafana        | http://localhost:13000 |
| Prometheus     | http://localhost:19093 |
| Collector Health | http://localhost:13133 |

---

## 全体アーキテクチャ

```
grpc / rest / bff
↓
OpenTelemetry Collector
↓
Prometheus（Metrics）
Loki（Logs）
Tempo（Traces）
↓
Grafana（可視化）
```

---

# 1. Grafana（メインUI）

Grafana はすべての観測データ（Metrics / Logs / Traces）を統合的に確認するためのUIである。

## 1.1 基本操作

### Explore 画面
- 左メニュー → `Explore`
- データソースを切り替えて利用する

| 種別   | データソース |
|--------|--------------|
| Metrics | Prometheus |
| Logs    | Loki |
| Traces  | Tempo |

---

## 1.2 Metrics の確認（Prometheus）

### 基本クエリ
```
Explore → Prometheus → Code
```

```promql
up
```

* 各サービスの稼働状態を確認
* `1` = 正常 / `0` = 異常

### リクエスト量の確認

```promql
rate(http_server_requests_seconds_count[5m])
```

---

## 1.3 Logs の確認（Loki）

### 全ログ表示

```logql
{job=~".+"}
```

### 特定サービスのログ

```logql
{service_name="grpc-backend"}
```

### エラーログ抽出

```logql
{service_name="grpc-backend"} |= "error"
```

---

## 1.4 Traces の確認（Tempo）

### 手順

1. Explore
2. Data source → `Tempo`
3. Search タブを使用

### フィルタ例

* `service.name = grpc-backend`
* 時間範囲: Last 15 minutes

### 確認できる内容

* リクエストの処理フロー
* Spanごとの処理時間
* ボトルネック箇所
* エラー発生箇所

---

## 1.5 実務での調査フロー

### ケース：エラー・遅延の調査

1. Metrics で異常検知
2. Logs で原因特定
3. Traces で詳細分析

```
Metrics → Logs → Traces
```

---

# 2. Prometheus（状態確認・デバッグ用）

Prometheus はメトリクス収集基盤であり、Grafana の裏側で動作している。

## 2.1 Targets 確認（最重要）

```
Status → Targets
```

### 確認ポイント

| 状態   | 意味   |
| ---- | ---- |
| UP   | 正常   |
| DOWN | 取得失敗 |

---

## 2.2 主な用途

* メトリクスが取得できているかの確認
* scrape エラーの確認
* Grafanaに表示されない原因の切り分け

---

## 2.3 クエリ確認
```
Explore → Prometheus → Code
```


```promql
up
```

Grafanaと同様の確認が可能。

---

# 3. OpenTelemetry Collector（Health）

Collector はデータの集約・転送を担うコンポーネントである。

## 3.1 ヘルスチェック

```
http://localhost:13133
```

### 正常レスポンス

```json
{
  "status": "Server available"
}
```

---

## 3.2 役割

* アプリケーションからのデータ受信
* Metrics / Logs / Traces の振り分け
* 各バックエンドへの転送

---

## 3.3 注意点

* このエンドポイントは **health check専用**
* メトリクスは別ポート（例: 9464）で提供される

---

# 4. 実践的な使い分け

## 4.1 開発時

* Grafana → Logs
* リアルタイムログ確認

---

## 4.2 パフォーマンス確認

* Grafana → Metrics
* `rate()` などで負荷確認

---

## 4.3 障害調査

1. Prometheus → 異常検知
2. Loki → エラーログ確認
3. Tempo → トレース分析

---

# 5. 今後の改善ポイント

* ダッシュボードの作成
* アラート設定（例: Slack通知）
* trace_id による Logs / Traces 連携強化

---

# まとめ

| ツール            | 役割      |
| -------------- | ------- |
| Grafana        | 可視化・分析  |
| Prometheus     | メトリクス収集 |
| Loki           | ログ収集    |
| Tempo          | トレース収集  |
| OTel Collector | データ集約   |

Observability の基本は以下の流れである。

```
Metrics → Logs → Traces
```

この流れを軸に調査を行うことで、効率的な原因特定が可能になる。
