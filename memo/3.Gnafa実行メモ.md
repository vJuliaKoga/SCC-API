# 4. k6 実行

## 共通ルール（重要）

* **run_id は必ずユニーク**
* 同じ run_id の再利用は禁止
* 1 run = 1 execution

---

## 環境変数

```bash
export BASE_URL=http://localhost:19090
export GIT_SHA=$(git rev-parse --short HEAD)
```

---

## 4.1 users-read（REST）

```bash
RUN_ID=local-$(date +%s)-users-read-rest

k6 run \
    --tag run_id=$RUN_ID \
    --tag git_sha=$GIT_SHA \
    --tag scenario=users-read \
    --tag call_mode=rest \
    --tag api=users-read \
    --out opentelemetry \
    k6/users-read.js
```

# 5. k6 実行結果の記録（最重要）

k6 実行後に表示される summary を記録する。

## 記録する値

* `http_req_duration p(95)`
* `http_reqs`
* `http_req_failed`
* `checks`

例:

```text
http_req_duration........: p(95)=12.3ms
http_reqs................: 10
http_req_failed..........: 0.00%
checks_succeeded.........: 100%
```

| --------- | ---------- | ---- | ---- | ------- | ---------- |
| local-xxx | users-read | rest | 12ms | 100     | 0%         |

# 7. Grafana の見方

## 使用目的

Grafana は「比較値の取得」ではなく、以下の確認用途に使う。

* run が記録されているか
* タグ（run_id / api / call_mode）が正しいか
* checks が成功しているか
* trace がつながっているか
* logs が出ているか

---

## 確認クエリ（これだけ使う）

```promql
sum by (check, condition, run_id, api, call_mode, scenario) (
    last_over_time(
        k6_checks_total[1h]
    )
)
```

---

## 見るポイント

* `condition="nonzero"` → 成功
* `condition="zero"` → 異常混入

---

# 8. トラブルシュート

## 8.1 チェック成功率がおかしい

原因:

* `condition="zero"` が混入

対応:

* その run は除外

---

## 8.2 Grafana が No data

原因:

* 変数フィルタ
* メトリクス名不一致

対応:

* raw クエリで確認
* 変数を使わない

---

## 8.3 rest-03 のような異常 run

対応:

* 比較対象から除外
* 再実行して新しい run_id を使う

---

## 8.4 データが混ざる

原因:

* 同じ run_id を再利用

対応:

```bash
RUN_ID=local-$(date +%s)-xxx
```

---

# 9. ベストプラクティス

## 絶対守るルール

* run_id は毎回変える
* 同一 run_id の再実行禁止
* 比較値は k6 summary を使う

---

# 10. 最短フロー（実務用）

1. BFF を REST / gRPC で起動
2. k6 実行（run_id ユニーク）
3. k6 summary をコピペ
4. 表に記録
5. Grafana で raw checks を確認
6. 異常 run を除外
7. 比較まとめる

---
