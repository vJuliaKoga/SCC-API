## 0. 目的

このプロジェクトでは、**BFF の外向け API を k6 で実行し、BFF 内部の `app.call-mode` を `rest` / `grpc` に切り替えて比較**する。

比較対象 API は以下。

- `GET /api/users/{id}`
- `POST /api/orders`

BFF の接続先は以下を利用する。

```text
http://localhost:19090
```

---

## 1. 前提条件

以下を事前に用意しておくこと。

- k6 がインストール済みであること
  - [k6](https://github.com/grafana/k6/releases)
  - またはPowershellで`winget install GrafanaLabs.k6`
- BFF が起動できること
- REST / gRPC backend が起動できること
- 先に通常の実行確認が完了していること

### 補足

k6 の比較は、**BFF の外部 API は同じまま、内部の接続方式だけを変える**のが目的。
そのため、比較時は **叩く URL・payload・スクリプト・実行条件を揃える**こと。

## 4. 実施前の確認順

最初は **smoke から実行**する。

### 確認順

1. backend 起動
2. BFF 起動
3. smoke 実行
4. benchmark 実行

いきなり benchmark を流さず、先に smoke で正常疎通を確認してから benchmark に進むこと。

---

## 5. smoke 実行例

### users API

```shell
k6 run k6/scripts/smoke/users-get.js
```

### orders API

```shell
k6 run k6/scripts/smoke/orders-post.js
```

### 確認観点

- ステータスコードが想定通りであること
- エラーにならないこと
- 閾値違反が出ていないこと
- 想定したレスポンスが返ること

---

## 6. benchmark 実行例

### users API

```shell
k6 run k6/scripts/benchmark/users-get.js
```

### orders API

```shell
k6 run k6/scripts/benchmark/orders-post.js
```

### 補足

benchmark は負荷をかけるため、**backend / BFF のログや CPU 使用率が大きく動く**ことがある。
ローカル比較では、他アプリをなるべく閉じた状態で行うこと。

---

## 7. BASE_URL を明示する場合

デフォルトでは `http://localhost:19090` を利用する。
変更したい場合は環境変数で上書きできる。

```shell
k6 run -e BASE_URL=http://localhost:19091 k6/scripts/smoke/users-get.js
```

### 注意

この比較では通常、**BFF を叩く前提**なので、比較用途では `BASE_URL` は `19090` のままにする。
別ポート指定は動作確認や個別検証のときのみ使用する。

---

## 8. userId を変える場合

### `GET /api/users/{id}` の対象 ID を変えたい場合

```shell
k6 run -e USER_ID=u001 k6/scripts/smoke/users-get.js
```

### 注意

比較時は REST / gRPC で **同じ ID を使う**こと。
片方だけ違うデータを参照すると、比較結果がぶれる可能性がある。

---

## 9. order payload を変える場合

`POST /api/orders` の payload は環境変数で上書きできる。

```shell
k6 run -e ORDER_USER_ID=u001 -e ITEM_CODE=BOOK-001 -e QUANTITY=1 k6/scripts/smoke/orders-post.js
```

### 注意

orders API は payload 差分でレスポンス時間が変わる可能性があるため、比較時は以下を揃える。

- `ORDER_USER_ID`
- `ITEM_CODE`
- `QUANTITY`

## 12. 比較時の注意点

比較を成立させるため、以下は揃えること。

- 同じ BFF エンドポイントを叩く
- 同じ payload を使う
- 同じ k6 スクリプトを使う
- 同じマシン条件で実行する
- 変更するのは `call-mode` だけにする

### 追加で意識すること

- 実行順の影響を見るため、可能なら複数回実施する
- 1回だけで結論を出さない
- benchmark 前に不要プロセスを止める
- 比較中はソース修正や設定変更を混ぜない

---

## 13. 最初に見る指標

k6 実行結果では、まず以下を見る。

- `http_req_duration`
- `http_req_failed`
- `p(95)`
- `p(99)`
- `iteration_duration`

### 最低限確認する値

- `p50`
- `p95`
- `p99`
- error rate

### 見方の目安

- `http_req_failed`
  エラー率確認用。まず 0% に近いことを確認する。

- `http_req_duration`
  レスポンス時間の中心指標。比較の主対象。

- `p95` / `p99`
  遅い側のばらつきを見る。平均だけでなく、後ろの重さを見る。

- `iteration_duration`
  1反復全体の時間。API 単体より少し広い観点の確認に使う。

---

## 14. 実行結果の見方

比較時は、単純な最速値だけでなく以下を確認する。

### まず見る順序

1. エラーが出ていないか
2. `p95` が大きく悪化していないか
3. `p99` の跳ね方に差がないか
4. 平均値だけでなく分位点も安定しているか

### 判断時の注意

平均値だけ良くても、`p95` や `p99` が悪い場合は、体感性能として不安定なことがある。
そのため、**平均よりもまず `p95` / `p99` を優先して確認**する。

---

## 15. トラブルシュート

### k6 コマンドが見つからない

```shell
k6 version
```

で確認する。
認識しない場合は、k6 が未インストールか Path 未設定の可能性がある。

---

### BFF に接続できない

以下を確認する。

```shell
Invoke-RestMethod http://localhost:19090/health
netstat -ano | findstr :19090
```

確認ポイント

- BFF が起動しているか
- `19090` で LISTEN しているか
- `app.call-mode` の設定が想定通りか

---

### smoke が失敗する

benchmark の前に、まず以下を確認する。

- backend が起動しているか
- BFF が起動しているか
- 対象 API を手動実行して成功するか
- 環境変数の指定ミスがないか

---

### gRPC 比較のはずが REST を呼んでいる

`bff/src/main/resources/application.yml` を確認する。

```yaml
app:
  call-mode: grpc
```

保存後、**BFF を再起動**すること。
設定変更だけでは反映されない。

---

## 16. 補足

- 比較用途では、まず smoke で疎通確認してから benchmark に進む
- 比較条件を崩さないことが最重要
- 実行結果はファイル保存して REST / gRPC を並べて確認すると見やすい
- 可能であれば 1 回ではなく複数回測定し、傾向で判断する
